### Path-traversal safety

CLAUDE.md hard rule: "Do not trust paths from arguments without
bounds + `../`-traversal check (relevant when an arg points at
a symlink chain that escapes the intended scope)."

Reviewed: darshini accepts a single path from argv and passes
it to syscalls without sanitization. The kernel resolves
`../` and symlinks. There is no "intended scope" boundary —
darshini is a general-purpose `ls`-equivalent. Same trust model
as `ls` / `cp` / `cat`.

Specific cases reviewed:
- **`git_find_root` walks up**: stops at the filesystem root or
  the first `.git/` directory. Bounded walk; no infinite loop
  even on symlink loops because each iteration shrinks the
  path string. ✓
- **`path_join(dir, name)` in compute_decor / tree.cyr**:
  `name` comes from `getdents64` which returns basenames
  (never `..` or paths with `/`). Concatenation is safe. ✓
- **`.git/index` / `.gitignore` open**: paths constructed from
  the verified-existing repo root. No untrusted input on the
  read path. ✓
- **mime magic-byte probe**: opens the listed entry by its full
  path (also `path_join` from listing dir + basename). Same
  guarantee. ✓

**Finding (INFO-3)**: path-traversal is not an exploitable
vector. Same trust boundary as the OS itself. The hard-rule
language in CLAUDE.md is precautionary; doesn't apply to a
general-purpose listing tool.

### External-format parser robustness

#### `_git_parse_index` (git.cyr)

- Validates `len >= 12` before reading header.
- Validates `DIRC` signature (4 bytes).
- Validates version == 2 (rejects v3/v4, returning -1).
- Per-entry: validates `pos + 62 <= len` and
  `name_start + name_len <= len` before reading. Malicious or
  truncated indexes return -1 without OOB read.

#### `_git_parse_ignore` (git.cyr)

- Line-by-line walk bounded by `len`. No buffer arithmetic
  beyond `pos <= len`.
- Comment / negation / blank lines filtered defensively.

#### `_mime_for_magic_bytes` (mime.cyr)

- Each signature has an explicit `len >= N` guard before byte
  access. Inputs smaller than the shortest signature (2 bytes)
  short-circuit to 0.

#### `_git_parse_index` malicious-input case (deeper look)

Considered: a `.git/index` file with `entry_count = 2³¹`
(claimed huge entry count). The loop walks `i < n`; if the
buffer runs out, we return -1 early. No quadratic blowup
because each iteration consumes at least 63 bytes of input.

Considered: an entry with `name_len = 0xFFF` (the "long name"
sentinel). The fallback scan walks `k < len` until NUL or EOF.
Bounded by `len`. ✓

**Finding (INFO-4)**: parsers are length-bounds-checked. No
trust-the-attacker reads.

### Allocation patterns

- `compute_decor` allocs ~200-400 bytes per entry (stat buf,
  path_join, color/icon/mime/git slots). For 10 000 entries
  that's ~3 MB.
- Initial heap is 2 MB (`alloc.cyr` default) but `brk(2)`
  extends on demand. No OOM in practice through 8k+ entries
  (verified against `/usr/bin`).

**Finding (MINOR-1)**: per-listing alloc footprint is N × O(1)
not amortized. A bump-arena-per-listing pattern would let the
heap reset between invocations, but invocations are
process-shot anyway so the heap dies with the binary. Not a
real concern.

### Bounds / overflow review

- `format_size_human` boundary constants (`1024`, `1048576`,
  `1073741824`, `1099511627776`) all fit in i64. The largest
  is 2⁴⁰; i64 max is 2⁶³-1. ✓
- `_be_u32` returns a value packed into i64; the OR-shift can't
  overflow because each byte contributes at most 8 bits.
- `pos + entry_total` in `_git_parse_index` is i64 + i64. Total
  index file size capped at 4 MB (the slurp cap); no overflow
  in realistic data.

**Finding (MINOR-2)**: `format_mtime` assumes a 4-digit year
via `_chrono_w4`. Year ≥ 10000 would overflow the buffer.
Triggers in ~8 000 years; not a real concern for v1.0. Document.

## Step 4 — External research

### What `eza` does that darshini deliberately doesn't

Cross-checked against the [`eza` v0.18 manpage](https://github.com/eza-community/eza/blob/main/man/eza.1.md)
plus the roadmap's "Out of scope for v1.0" block. The
deliberate-omissions list:

| eza feature                          | darshini scope decision |
|--------------------------------------|-------------------------|
| Gitignore-aware filtering (`--git-ignore`) | OUT. `--git` shows status; filtering is `find`'s job. |
| Extended attributes (`@`)            | OUT. xattr surface is platform-divergent + niche. |
| ACLs (`+`)                           | OUT. Same reason. |
| `--sort=size` / `--sort=mtime` / `--sort=name` | OUT. Alphabetical-only at v1.0. |
| Recursive non-tree (`-R`)            | OUT. Use `-T --level N`. |
| Configurable via dotfile (`EZA_COLORS` / `~/.config/eza`) | OUT. Flag surface IS the contract. |
| Per-icon color override (LS_COLORS)  | OUT. Mode-derived color per ADR 0001. |
| File-size sort suffixes (`-b`)       | OUT. Single human/decimal mode. |
| `--no-permissions` / `--no-filesize` | OUT. `-l` is on/off, no column-level toggles. |
| Header row (`--header`)              | OUT. Adds dependency on column-width post-compute. |
| Hyperlink output (`--hyperlink`)     | OUT. Terminal-emulator divergent. |
| Mounts column (`--mounts`)           | OUT. Linux mount-table parsing is M9-scope creep. |
| Total-size summary at end            | OUT. Same scope reason. |
| `-F` (classify with `/`, `*`, `@`, `\|`) | **OUT for v1.0** (post-v1 candidate — surfaced by user's dotfile alias `lfiles=eza -F`). |
| `-d` (list dirs themselves)          | **OUT for v1.0** (post-v1 candidate — same source: `ldir`). |
| Glob filtering (`-I`)                | OUT. `find` + `xargs` is the right tool. |
| `--smart-group` (Unix group display) | OUT. Long-format columns frozen at perms/size/mtime per ADR 0002 sibling text. |

**Finding (INFO-6)**: the "what we deliberately don't do" list
is consistent with the roadmap. Two recently-surfaced
candidates (`-F`, `-d`) noted as v1.1 work if the maintainer
wants full eza removal from the dotfile aliases.

### Cross-reference with kriya

Per CLAUDE.md "Key Principles", darshini is "sibling, not
replacement" of kriya's `ls`. kriya is the BusyBox-style `ls`
for scripts. darshini is the human-eyes-on-iron variant.

The non-overlap is intentional:
- kriya's `ls`: stable bytes-out for pipes, minimal flag set,
  shell-script-safe.
- darshini: rich TTY output, larger flag set, opinionated
  defaults (icons + color on TTY).

Both ship; users alias whichever they prefer. The dotfile work
this session reflects this — `ll` / `l` → darshini, scripts
keep using kriya's `ls`.
